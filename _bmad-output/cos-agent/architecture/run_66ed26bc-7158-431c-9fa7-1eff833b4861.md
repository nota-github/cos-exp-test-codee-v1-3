## Source Context

- Supporting source: brief `run_1420877c-a28e-4cd1-830d-b3b890a0789e.md`
- Current workspace state: 승인된 문서와 `README.md`만 존재하는 greenfield 프론트엔드 시작점이다. 따라서 본 문서는 기존 구현 제약을 계승하기보다, v0 구현이 흔들리지 않도록 최소 기술 선택과 경계 조건을 명시한다.

## Architecture Overview
이 제품은 `브라우저 단일 화면 + 로컬 상태 저장`만으로 완결되는 초경량 체크인 앱이다. PRD의 핵심은 `한 화면`, `정확히 3개의 기분/할 일 구조`, `새로고침 후 복원`, `모바일 자연스러움`, `백엔드 없음`이며, 아키텍처도 이 다섯 가지를 가장 우선하는 방향으로 고정한다.

권장 기술 베이스라인은 다음과 같다.

- 클라이언트 전용 정적 웹 앱
- `Vite + TypeScript + semantic HTML + plain CSS`
- 라우터 없음
- 백엔드/API/auth 없음
- 전역 상태 라이브러리 없음
- 브라우저 `localStorage` 기반 영속화

이 선택의 이유는 범위가 매우 작고 화면이 하나뿐이기 때문이다. v0에서 프레임워크 오버헤드를 늘리기보다, 타입이 있는 상태 모델과 분리된 모듈 경계로 구현 복잡도를 통제하는 편이 더 적합하다. 단, UI 렌더링 계층과 저장 계층을 분리해 두어 향후 React 같은 컴포넌트 프레임워크로 옮기더라도 데이터 계약은 유지되도록 설계한다.

PRD 추적 관점에서의 핵심 번역은 다음과 같다.

- `FR-1`, `FR-6`: 날짜, 기분, 할 일, 메모는 첫 렌더에서 모두 보여야 하며 추가 화면 이동 없이 사용 가능해야 한다.
- `FR-2`, `FR-3`, `FR-4`: 기분은 단일 선택 3종, 할 일은 고정 3행, 메모는 독립된 다중 행 입력으로 구현해야 한다.
- `FR-5`: 상태는 앱 전체 단위로 저장/복원되어야 하며 저장 실패 시에도 앱이 깨지면 안 된다.
- `FR-7`: 시각 구조는 밝고 차분한 대시보드 톤을 유지하되, 상태 변화와 입력 경계가 명확해야 한다.
- `FR-8`: 런타임은 브라우저만 전제하며 외부 서비스 의존성을 추가하지 않는다.

## Major Components

### 1. App Shell and Responsive Layout
앱 셸은 단일 페이지 레이아웃과 섹션 위계를 담당한다. 상단에는 오늘 날짜와 가벼운 헤더 문구를 배치하고, 그 아래에 기분 선택, 할 일 3개, 메모를 같은 화면 흐름으로 배치한다.

구현 제약은 다음과 같다.

- 데스크톱에서는 카드형 대시보드 레이아웃을 사용하되, 정보가 흩어져 보이지 않도록 최대 2열 수준으로 제한한다.
- 모바일에서는 세로 스택으로 재배치하되, 핵심 기능을 아코디언, 탭, 모달 뒤에 숨기지 않는다.
- `한 화면` 요구는 `한 라우트/한 페이지 안에서 모든 핵심 기능 제공`으로 해석한다. 매우 작은 모바일 viewport에서 약간의 세로 스크롤은 허용하지만, 정보 밀도를 무리하게 높여 무가독 UI를 만들면 안 된다.

이 컴포넌트는 이후 구현에서 스타일 토큰, 카드 간격, 모바일 브레이크포인트를 통합 관리하는 기준점이 된다.

### 2. Date Display Module
날짜 모듈은 사용자 기기 기준의 오늘 날짜를 표시한다. 이 모듈은 서버 시간이 아니라 브라우저 환경의 현재 날짜를 사용해야 하며, 첫 렌더에서 즉시 보여야 한다.

구현 제약은 다음과 같다.

- 날짜는 읽기 전용 표시 요소다.
- 포맷팅은 브라우저 locale 기반 또는 한국어 고정 포맷 중 하나로 일관되게 처리하되, 앱 전체에서 하나의 유틸리티만 사용한다.
- 저장된 체크인 내용과 무관하게 표시 날짜는 항상 `현재 기기 날짜`를 기준으로 계산한다.

### 3. Mood Selector Module
기분 선택은 정확히 3개의 옵션을 동시에 노출하는 단일 선택 UI다. 라디오 그룹 또는 토글 버튼 그룹으로 구현할 수 있지만, 키보드와 터치 조작이 모두 자연스러워야 한다.

구현 제약은 다음과 같다.

- 선택지는 상수 배열로 관리한다.
- 각 선택지는 `stable key`와 `display label`을 분리한다.
- 한 번에 하나만 선택 가능해야 한다.
- 선택/비선택 상태가 배경색, 테두리, 텍스트 대비 중 최소 2개 이상의 시각 신호로 구분되어야 한다.

기분 3종의 문구와 아이콘 여부는 아직 제품 확정 사항이 아니므로, 문자열은 한 파일 또는 상수 블록에 모아 둬야 한다. UI 마크업과 저장 포맷 안에 하드코딩된 한국어 라벨을 직접 중복 삽입하면 이후 수정 비용이 커진다.

### 4. Todo Slots Module
할 일 영역은 `고정 3행` 구조를 가진다. 각 행은 체크박스 1개와 텍스트 입력 1개로 구성되며, 독립적으로 수정되어야 한다.

구현 제약은 다음과 같다.

- 내부 데이터 구조는 배열을 사용하되, 정확히 3개 항목만 생성한다.
- 각 항목은 `id`, `text`, `checked`를 가진다.
- 추가/삭제/정렬 기능은 구현하지 않는다.
- 체크 영역과 텍스트 입력 영역은 터치 환경에서 오작동하지 않도록 충분히 분리한다.

이 모듈은 범위상 단순하지만, 향후 스토리 분할을 위해 `행 렌더링`, `입력 이벤트 처리`, `영속화 트리거`를 분리 가능한 구조로 두는 것이 좋다.

### 5. Memo Module
메모는 자유 입력용 다중 행 텍스트 영역이다. 할 일과는 분리된 독립 카드 또는 독립 섹션으로 보여야 하며, 짧은 메모를 빠르게 남기는 데 초점을 둔다.

구현 제약은 다음과 같다.

- 기본 HTML `textarea`를 사용한다.
- 자동 높이 확장 여부는 선택 사항이지만, 기본 높이만으로도 작성 시작이 불편하지 않아야 한다.
- 할 일 입력 필드와 시각적으로 다른 정보 덩어리로 보여야 한다.

### 6. Checkin State Store
앱의 실제 중심은 UI가 아니라 상태 저장 계약이다. 날짜, 기분, 할 일, 메모를 각각 따로 저장하는 방식보다, 체크인 전체를 하나의 상태 객체로 다루는 것이 초기화/복원/마이그레이션에 유리하다.

권장 상태 구조는 다음과 같다.

```ts
type MoodKey = string | null;

type TodoItem = {
  id: 'todo-1' | 'todo-2' | 'todo-3';
  text: string;
  checked: boolean;
};

type CheckinState = {
  schemaVersion: 1;
  localDate: string | null;
  mood: MoodKey;
  todos: [TodoItem, TodoItem, TodoItem];
  memo: string;
  updatedAt: string | null;
};
```

이 구조에서 중요한 점은 다음과 같다.

- `schemaVersion`은 추후 저장 포맷 변경 대비용이다.
- `localDate`는 저장 시점의 로컬 날짜를 기록한다. 이는 당장 히스토리를 제공하기 위한 필드는 아니고, 날짜 변경 시 동작 결정을 나중에 분기하기 위한 안전장치다.
- `updatedAt`은 디버깅과 미래 확장 여지를 위해 보존한다.

### 7. Persistence Adapter
영속화는 `localStorage` 접근을 직접 UI 곳곳에서 호출하지 않고, 별도 어댑터로 감싸는 구조가 적합하다. 이렇게 해야 파싱 실패, 저장 실패, 스키마 검증, 기본값 복구를 한 군데서 처리할 수 있다.

권장 저장 규칙은 다음과 같다.

- 저장 키는 단일 키 하나를 사용한다. 예: `mini-checkin:v1`
- 저장 값은 JSON 문자열 하나로 직렬화한다.
- 앱 시작 시 `default state -> storage read -> parse -> validate -> hydrate` 순서로 초기화한다.
- 텍스트 입력은 짧은 debounce를 두고 저장하고, 체크박스/기분 선택은 즉시 저장해도 된다.

단일 키 전략을 권장하는 이유는 다음과 같다.

- 한 번의 읽기로 전체 상태를 복원할 수 있다.
- 기분, 할 일, 메모 사이의 저장 시점 불일치를 줄일 수 있다.
- 스키마 버전 업 시 마이그레이션 지점이 하나로 고정된다.

## Data, State, or Persistence Concerns

### Single Source of Truth
UI 상태와 저장 상태는 동일한 `CheckinState` 구조를 공유해야 한다. 렌더링은 항상 메모리상의 현재 상태를 기준으로 하고, 저장은 그 상태를 직렬화하는 방식으로 동작해야 한다. DOM 값을 진실 소스로 삼고 필요할 때마다 다시 읽는 방식은 작은 앱에서도 버그를 만들기 쉽다.

### Initialization Rules
초기 상태는 항상 안전한 기본값으로 시작해야 한다.

- 기분: `null`
- 할 일 3개: 빈 문자열 + 체크 해제
- 메모: 빈 문자열
- 날짜: 현재 기기 날짜 계산값

저장 데이터가 없거나 JSON 파싱에 실패하거나 구조가 기대와 다르면, 앱은 오류를 던지지 말고 기본 상태로 복귀해야 한다.

### Date Boundary Handling
PRD에서 아직 미해결인 가장 중요한 설계 포인트는 `날짜가 바뀌었을 때 기존 로컬 저장 내용을 어떻게 처리할 것인가`이다. 이 문서에서는 이를 조용히 확정하지 않는다. 대신 구현이 다음 형태를 따르도록 제약한다.

- 상태에는 저장 당시 `localDate`를 남긴다.
- 초기 상태 복원은 `resolveInitialState(currentDate, savedState)` 같은 분리된 정책 함수에서 결정한다.
- 이 정책 함수 바깥의 UI 코드가 날짜 비교 로직을 직접 가지면 안 된다.

이렇게 하면 이후 제품 결정이 다음 세 가지 중 어느 쪽으로 나와도 구현 범위를 국소화할 수 있다.

1. 날짜가 바뀌어도 마지막 입력을 그대로 복원
2. 날짜가 바뀌면 초기화
3. 날짜별로 분리 저장

### Storage Failure Handling
브라우저 설정, 사파리 프라이빗 모드류 제약, 사용자 저장소 정리 등으로 `localStorage`가 실패할 가능성은 낮지만 존재한다. v0 범위에서 필수 요구는 `저장 실패 시 앱이 깨지지 않는 것`이다.

최소 요구 동작은 다음과 같다.

- 읽기 실패 시 기본 상태로 동작
- 쓰기 실패 시 현재 세션 메모리 상태는 유지
- 실패를 조용히 삼키되, 필요하면 개발 단계에서만 콘솔 경고를 남긴다

### Mood Label Variability
기분 옵션 라벨은 아직 제품 문구가 확정되지 않았으므로 저장 값은 `label`이 아니라 `key`를 기준으로 유지해야 한다. 예를 들어 `"good"`, `"neutral"`, `"tired"` 같은 내부 키를 쓰고, 한국어 라벨은 표시 전용으로 매핑하는 방식이 적합하다.

## Integration and Runtime Constraints

### Runtime Boundaries
- 브라우저만 런타임으로 가정한다.
- 네트워크 연결을 필수로 요구하면 안 된다.
- 외부 API, 계정 시스템, 분석 SDK, 원격 설정, 서버 시간 의존성을 추가하면 안 된다.

### Build and Launch Constraints
- 개발 모드는 로컬 dev server에서 즉시 실행 가능해야 한다.
- 배포 관점에서는 정적 파일 호스팅만으로 충분해야 한다.
- SSR, Node 서버, 데이터베이스, 백엔드 프로세스는 v0 범위에서 금지한다.

### UX Runtime Constraints
- 첫 렌더에서 핵심 섹션 전부가 보여야 한다.
- 저장을 위해 별도 제출 버튼을 필수 요구로 두지 않는다. 이 제품의 성격상 자동 저장이 더 자연스럽다.
- 단, 자동 저장을 선택하더라도 상태 변화 피드백은 지나치게 요란하면 안 된다. "저장됨" 토스트 남발은 초경량 UX와 맞지 않는다.

### Responsive and Accessibility Constraints
- 모바일 우선으로 레이아웃을 설계하고, 넓은 화면에서 카드 배치를 확장한다.
- 터치 타깃은 충분한 높이와 간격을 가져야 한다.
- 포커스 링을 제거하지 않는다.
- 각 입력 요소는 시각 라벨 또는 접근 가능한 이름을 가져야 한다.
- 색상만으로 상태를 구분하지 말고, 선택 테두리/체크 표시/텍스트 대비를 함께 사용한다.

### Styling System Constraints
시각 톤은 `밝고 차분한 대시보드`가 기준이므로, 스타일링은 처음부터 토큰화하는 편이 낫다.

- CSS 변수로 배경, 카드, 테두리, 강조색, 성공/선택 상태색을 정의한다.
- 입력창과 버튼은 테두리, 배경, focus 상태가 분명히 구분되어야 한다.
- 과한 그림자, 과한 애니메이션, 정보 밀도를 해치는 장식은 피한다.

## Implementation Risks and Tradeoffs

### 1. Date Reset Policy Is Still a Product Decision
가장 큰 리스크는 날짜 변경 시 복원 정책이 아직 확정되지 않았다는 점이다. 이 결정은 저장 키 구조와 첫 진입 동작에 직접 영향을 준다. 따라서 구현은 정책 함수를 분리한 상태에서 진행해야 하며, 이 이슈를 무시한 채 저장 구조를 UI 코드에 박아 넣으면 이후 수정 비용이 커진다.

### 2. Single-Page Density vs Mobile Readability
`한 화면에 다 보여야 한다`는 요구를 `어떤 모바일 높이에서도 무조건 no-scroll`로 해석하면 UI 품질이 급격히 나빠질 수 있다. 본 아키텍처는 이를 `한 페이지 안에서 즉시 접근 가능`으로 제한 해석하며, 작은 화면에서는 적당한 세로 흐름을 허용한다. 이는 UX 품질을 위한 의도적 tradeoff다.

### 3. Minimal Stack vs Future UI Abstraction
`Vite + TypeScript + plain CSS`는 지금 범위에 가장 적합하지만, 추후 기능이 늘어나면 컴포넌트 프레임워크 부재가 생산성 한계로 느껴질 수 있다. 이를 완화하기 위해 렌더링 코드와 상태/저장 코드를 분리하고, 기분/할 일/메모를 독립 모듈처럼 다루는 구조가 필요하다.

### 4. Autosave Simplicity vs Write Frequency
자동 저장은 이 제품에 가장 잘 맞는 UX지만, 입력마다 저장이 발생한다. 다만 payload가 매우 작기 때문에 성능 문제 가능성은 낮고, 텍스트 입력에 짧은 debounce를 두면 충분하다. 복잡한 저장 큐나 비동기 스토리지는 v0에서 과하다.

### 5. Fixed Todo Count Improves Clarity but Limits Extensibility
정확히 3개의 할 일만 보이게 하는 구조는 PRD와 UX에 매우 잘 맞고 구현도 단순하다. 대신 이후 동적 추가를 원하면 데이터 모델과 UI 구조를 수정해야 한다. 따라서 내부 모델은 배열 기반으로 두되, 렌더링 개수만 현재 3개로 고정하는 방식이 가장 안전하다.

### 6. localStorage Reliability Is Convenience-Level Only
이 앱의 저장은 동일 브라우저 내 편의 기능이지 영구 보관 수단이 아니다. 사용자가 브라우저 데이터를 지우거나 저장소 제한이 걸리면 데이터는 사라질 수 있다. 이는 비기능적 한계로 문서화되어야 하며, 백업/동기화 기대를 만들면 안 된다.

## Decisions Needed / Carry-Forward Notes
- 기분 3종의 실제 라벨과 시각 표현은 UX 단계에서 확정 필요
- 날짜 변경 시 복원 정책은 제품 결정 필요
- 자동 저장 외에 명시적 저장 피드백이 필요한지 여부는 UX 미세조정 사항

## Architecture Summary
v0는 `단일 페이지`, `고정된 소규모 상태 모델`, `버전 있는 단일 localStorage payload`, `모바일 우선 반응형 카드 레이아웃`으로 설계하는 것이 가장 적절하다. 이 구조는 PRD의 모든 핵심 요구를 직접 충족하면서도, 아직 확정되지 않은 날짜 정책과 기분 라벨 결정을 별도 경계로 분리해 이후 변경 비용을 최소화한다.
