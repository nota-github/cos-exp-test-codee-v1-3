# 오늘의 미니 체크인 - PRD v0

## Goals
- 사용자가 오늘의 상태를 한 화면에서 빠르게 기록할 수 있는 초경량 체크인 경험을 제공한다.
- 첫 진입 시 오늘 날짜, 기분 선택, 할 일 3개, 짧은 메모가 모두 즉시 보이도록 한다.
- 사용자가 같은 브라우저에서 새로고침하거나 다시 열어도 마지막 입력 상태를 그대로 이어볼 수 있게 한다.
- 데스크톱과 모바일 모두에서 깔끔하고 가벼운 대시보드 느낌을 유지한다.

## Non-Goals
- 사용자 계정, 로그인, 권한 관리
- 클라우드 동기화, 서버 저장, 백엔드 API 연동
- 할 일 개수의 동적 추가/삭제, 우선순위, 알림, 반복 일정
- 날짜별 히스토리, 통계, 회고 리포트, 공유 기능
- 협업, 댓글, 다중 사용자 기능

## Core Requirements

### Product Constraints
- 제품은 별도 백엔드 없이 바로 실행 가능한 단일 웹 앱 경험을 제공해야 한다.
- 저장은 브라우저 로컬 저장소만 사용해야 한다.
- 핵심 기능은 라우팅이나 별도 상세 화면 없이 한 페이지 안에서 제공되어야 한다.

### Functional Requirements
#### FR-1. 오늘 날짜 표시
- 첫 화면 진입 시 오늘 날짜가 사용자에게 즉시 보여야 한다.
- 날짜는 사용자의 현재 기기 기준 오늘 정보를 반영해야 한다.
- 날짜 영역은 다른 입력 요소에 묻히지 않도록 화면 상단의 인지 가능한 위치에 있어야 한다.

#### FR-2. 기분 선택
- 기분 선택지는 정확히 3개여야 한다.
- 세 선택지는 첫 화면에서 동시에 보여야 한다.
- 사용자는 한 번에 하나의 기분만 선택할 수 있어야 한다.
- 현재 선택된 기분은 시각적으로 분명히 구분되어야 한다.

#### FR-3. 할 일 3개 입력 및 체크
- 할 일 입력 행은 정확히 3개가 기본 화면에 항상 보여야 한다.
- 각 행은 텍스트 입력과 체크 상태를 함께 가져야 한다.
- 사용자는 각 할 일을 독립적으로 입력하고 체크/해제할 수 있어야 한다.
- v0 범위에서는 할 일 행의 추가, 삭제, 재정렬은 제공하지 않는다.

#### FR-4. 짧은 메모 작성
- 사용자는 한 화면 안에서 짧은 메모를 자유롭게 작성할 수 있어야 한다.
- 메모 입력 영역은 여러 줄 입력을 지원해야 한다.
- 메모는 할 일 입력과 구분되는 독립 영역으로 보여야 한다.

#### FR-5. 새로고침 후 상태 유지
- 기분 선택 상태, 할 일 3개의 텍스트, 각 체크 상태, 메모 내용은 브라우저에 로컬로 유지되어야 한다.
- 사용자가 같은 브라우저에서 페이지를 새로고침하거나 다시 열었을 때 마지막으로 저장된 상태가 복원되어야 한다.
- 저장된 데이터가 없을 때는 오류 없이 깔끔한 초기 상태를 보여야 한다.

#### FR-6. 한 화면 중심 경험
- 날짜, 기분, 할 일, 메모의 모든 핵심 기능은 하나의 화면 흐름 안에서 완료 가능해야 한다.
- 사용자가 핵심 기능을 쓰기 위해 추가 페이지 이동, 숨겨진 단계, 필수 모달 상호작용을 거치지 않도록 해야 한다.
- 모바일에서는 섹션이 세로로 재배치될 수 있지만, 동일한 정보와 기능 세트를 유지해야 한다.

#### FR-7. 시각적 명확성
- 전체 화면은 밝고 차분한 색조의 대시보드 느낌이어야 한다.
- 버튼, 선택 요소, 입력창은 경계와 상태 차이가 분명해야 한다.
- 모바일에서도 터치 가능한 크기와 읽기 쉬운 대비를 유지해야 한다.
- 섹션 간 정보 위계가 분명해야 하며, 화면이 과하게 복잡해 보이지 않아야 한다.

#### FR-8. 즉시 사용 가능성
- 사용자는 별도 가입이나 설정 없이 앱을 바로 사용할 수 있어야 한다.
- 핵심 경험은 로컬 실행만으로 성립해야 하며, 외부 서비스 연결을 전제하지 않아야 한다.

## User Flows or Core Requirements

### Flow 1. 첫 사용자의 오늘 체크인 시작
1. 사용자가 앱을 열면 오늘 날짜가 보인다.
2. 사용자는 세 가지 기분 중 하나를 선택한다.
3. 사용자는 보이는 세 개의 할 일 칸에 필요한 내용을 입력한다.
4. 완료한 항목이 있으면 해당 행을 체크한다.
5. 사용자는 짧은 메모를 작성한다.
6. 사용자는 별도 백엔드나 계정 없이 같은 화면에서 오늘 체크인을 마친다.

### Flow 2. 새로고침 또는 재방문
1. 사용자가 같은 브라우저에서 앱을 새로고침하거나 다시 연다.
2. 이전에 기록한 기분, 할 일 3개, 체크 상태, 메모가 복원된다.
3. 사용자는 이어서 수정하거나 그대로 확인할 수 있다.

### Flow 3. 모바일에서의 빠른 입력
1. 사용자가 모바일에서 앱을 연다.
2. 날짜, 기분, 할 일, 메모가 한 페이지 안에서 자연스러운 순서로 보인다.
3. 사용자는 작은 화면에서도 각 입력 요소를 무리 없이 탭하고 수정할 수 있다.

## Risks and Open Questions
- 기분 3종의 구체적인 라벨과 표현 방식은 아직 확정되지 않았다. 이후 UX 단계에서 선택지 문구와 시각 표현을 승인해야 한다.
- 날짜가 바뀌었을 때 기존 로컬 저장 데이터를 그대로 유지할지, 새 날짜 기준으로 초기화하거나 분리할지는 아직 제품 결정이 필요하다. 이 결정은 저장 키 구조와 첫 화면 동작에 영향을 준다.
- 브라우저 로컬 저장소는 사용자의 브라우저 설정이나 데이터 삭제 정책에 따라 초기화될 수 있다. 따라서 본 제품의 저장은 같은 브라우저 내 편의 기능으로 간주해야 하며, 영구 보관을 전제하면 안 된다.
- 매우 작은 모바일 화면에서는 한 화면 요구와 여백/가독성 요구가 충돌할 수 있다. 디자인 단계에서 정보 밀도와 터치 편의성의 균형을 확인해야 한다.
